# TCP协议
Transmission Control Protocol
传输控制协议

tcp

急性子 冲动快速       适合实时活动(打游戏,看直播)->丢一两帧不影响体验
## 1 它是干嘛的
负责两台电脑之间可靠地传送数据

### 那和FTP有什么关系?
FTP依赖TCP，没有TCP，FTP根本没法在两台电脑间传文件
TCP只会“裸奔传字节”，但它不知道什么是文件、文件夹、上传、下载；
FTP就是在TCP基础上，定好了一套「传文件专用规则」。

先分清角色:
TCP = 一条修好的空旷马路
只能帮两台电脑连通，能运输一堆二进制数据，但它不懂文件是什么，不会列目录、不会新建文件夹、不会删文件、不知道啥叫上传下载。
FTP = 专门拉文件的物流系统
它跑在 TCP 这条马路上，提前定好各种命令：登录、看文件列表、切换文件夹、上传、下载、删除、重命名。

###  只用TCP不传FTP，会有什么问题？
如果只靠 TCP 自己传数据：
你得自己写代码定义规则：怎么告诉对方我要传哪个文件、文件名是什么、大小多少、放到哪个文件夹。
没有统一标准，你写的程序别人连不上，每家都一套规矩，没法通用。
没有登录认证、没有权限管理，谁都能随便连、随便删文件，不安全。

有了 FTP 之后：全世界统一标准，所有软件都按 FTP 规则来，只要支持 FTP，就能互相传文件，不用自己从零造规则。

#### 也就是,tcp搭好基础环境,ftp制定行为规则
TCP：搭好基础环境负责两台电脑建立连接、修好通道、可靠传输数据，只负责通路，不管传的是什么内容
FTP：制定行为规则在 TCP 搭好的通道上，规定好传文件的一套标准动作：登录、看文件列表、上传、下载、删文件、建文件夹，大家都按统一规矩来

## 2 TCP怎么在两台电脑间传数据
两台电脑：A 电脑 → TCP 服务端B 电脑 → TCP 客户端
流程：
A 电脑开服务，监听某个端口，静静等着别人连；
B 电脑用 A 的 IP + 端口 主动找 A；
经过三次握手，两台电脑建立好专属通信通道；
之后两台电脑就可以互相发数据、发消息、传文件；
传完数据，用四次挥手断开连接。

## 3是不是感觉tcp在两台电脑传数据的实现方法和ftp差不同
### 1为什么你觉得它们实现方式很像？
因为：
都是一台当服务端、一台当客户端
都是IP + 端口连接
都是两台电脑建立通信通道
都是靠客户端主动连服务端
所以表面流程长得几乎一模一样，你感觉很正常。

### 2 差别到底在哪？
TCP 做的是：修路
只负责：
三次握手建连接
保证数据不丢、不乱、不错
只管把一堆二进制数据从 A 送到 B
TCP 根本不知道自己传的是文件、文字还是图片，它只管搬运字节。
FTP 做的是：在修好的路上按规矩办事
FTP 建立在TCP通路之上
它规定了：
怎么登录
怎么看文件夹
怎么上传文件
怎么下载文件
怎么删文件、改文件名
FTP 懂什么叫文件，TCP 不懂。

## 4 tcp能在服务器和客户端实现什么的传输
### 二进制数据
### 普通文字数据
聊天消息、指令、文本内容。
### 文件原始数据
图片、视频、文档、软件、压缩包的二进制源码，TCP原样帮你搬运。
### 网页数据
浏览器访问网页，网页的文字、图片、代码全是靠 TCP传。
### 各种协议的数据
FTP、HTTP、微信、QQ、邮箱 全都把自己的数据，交给 TCP 来帮它在两台电脑之间传输。

## 5 tcp和ftp一起使用
FTP 是坐在 TCP 身上干活的
TCP 负责修路、建通道、传数据；
FTP 负责在这条通道里定好传文件的规矩。

没有 TCP：FTP 根本连不上两台电脑，没法传任何东西，相当于没路走。
没有 FTP：TCP 只能瞎传一堆乱数据，不懂什么叫登录、看文件夹、上传下载，没法正经传文件。

### 实际工作过程（一起配合）
先由 TCP 帮两台电脑：建立连接、打通可靠通道；
然后 FTP 在这个 TCP 通道里，发各种命令：
登录、看文件列表、上传、下载、删文件；
真正的文件数据，也靠 TCP 稳稳传过去。

# 似乎全场最有用的...
TCP 只管搬运二进制；
FTP 只管规定二进制代表什么命令；
把二进制变成可读内容，是电脑软件和系统干的活。
